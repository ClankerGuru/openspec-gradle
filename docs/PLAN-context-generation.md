# Plan: Gradle-Powered Context Generation

## The Insight

AI coding assistants waste time discovering what Gradle already knows. Every time Copilot or Claude starts working on a Gradle project, they `grep` through build files, `find` source directories, and manually piece together the architecture. Meanwhile, Gradle has already resolved the full dependency graph, knows every source set, every module relationship, every applied plugin.

**We should extract this knowledge once, cache it, and hand it to the AI as a pre-built context file.**

---

## What Gradle Knows (API Surface)

### Project Structure
```kotlin
project.subprojects              // all submodules
project.rootProject              // root
project.projectDir               // where code lives
project.buildFile                // build.gradle.kts path
project.group / project.version  // coordinates
```

### Dependencies (the big one)
```kotlin
// Resolved dependency tree — external libs + inter-module
project.configurations["runtimeClasspath"].resolvedConfiguration
    .firstLevelModuleDependencies  // direct deps with transitives

// Inter-module dependencies  
project.configurations["implementation"].dependencies
    .filterIsInstance<ProjectDependency>()  // module → module links
```

### Source Sets
```kotlin
project.extensions.getByType<SourceSetContainer>().forEach { sourceSet ->
    sourceSet.name           // "main", "test", "integrationTest"
    sourceSet.allSource.srcDirs  // actual directories
    sourceSet.output.classesDirs // compiled output
}
```

### Applied Plugins (→ framework detection)
```kotlin
project.plugins.hasPlugin("org.springframework.boot")
project.plugins.hasPlugin("org.jetbrains.kotlin.jvm")
project.plugins.hasPlugin("com.android.application")
project.plugins.hasPlugin("org.jetbrains.compose")
// etc.
```

### Tasks
```kotlin
project.tasks.names  // all registered tasks
project.tasks.withType<Test>()  // test tasks
project.tasks.withType<Jar>()   // packaging
```

## What We Generate

### `openspecContext` task → `.openspec/context.md`

A single markdown file the AI reads at the start of any session. Cacheable — only regenerates when build inputs change.

```markdown

## Overview
- **Name:** my-app
- **Group:** com.example
- **Version:** 2.1.0
- **Build system:** Gradle 9.4 + Kotlin DSL
- **Language:** Kotlin 2.1.0, Java 17

## Modules

### :core
- **Path:** core/
- **Dependencies:** kotlin-stdlib, kotlinx-coroutines-core
- **Depended on by:** :api, :web, :cli

### :api  
- **Path:** api/
- **Dependencies:** :core, spring-boot-starter-web, jackson-module-kotlin
- **Source sets:** main (src/main/kotlin), test (src/test/kotlin)

### :web
- **Path:** web/
- **Dependencies:** :core, :api, spring-boot-starter-thymeleaf

## Module Graph
```
:core ←── :api ←── :web
  ↑
  └────── :cli
```

## Frameworks Detected
- **Spring Boot 3.2.4** (web, data-jpa, security)
- **Kotlin 2.1.0** (JVM target 17)
- **JUnit 5** (testing)

```

## Architecture Inference Rules

Based on detected plugins/dependencies, we can infer patterns:

| Detection | Inference |
|---|---|
| `spring-boot-starter-web` | REST API, controller/service pattern |
| `spring-boot-starter-data-jpa` | Entities, repositories, DB access |
| `spring-security` | Auth layer, security config exists |
| `flyway` or `liquibase` | Schema changes need migrations |
| `kotlin-serialization` | Uses @Serializable, not Jackson |
| `compose` (JB or Android) | Declarative UI, composable functions |
| `android.application` | Android app, activities/fragments |
| `protobuf` | gRPC/proto definitions, generated code |
| Multi-module + `:core` | Shared kernel pattern |
| `testcontainers` | Integration tests use Docker |
| `jooq` | Type-safe SQL, generated from schema |

These get written as "Architecture Hints" — practical guidance the AI can act on immediately.

## Caching Strategy

```kotlin
@CacheableTask
abstract class OpenSpecContextTask : DefaultTask() {

    // INPUTS — when these change, context regenerates
    @get:InputFiles
    @get:PathSensitive(PathSensitivity.RELATIVE)
    val buildFiles: ConfigurableFileCollection  // all build.gradle.kts
    
    @get:InputFiles  
    @get:PathSensitive(PathSensitivity.RELATIVE)
    val gradleProperties: ConfigurableFileCollection  // gradle.properties
    
    @get:InputFiles
    @get:PathSensitive(PathSensitivity.RELATIVE)
    val lockFiles: ConfigurableFileCollection  // dependency lock files
    
    @get:Input
    val resolvedDependencies: Provider<String>  // dependency hash

    // OUTPUT — cached result
    @get:OutputFile
    val contextFile: RegularFileProperty  // .openspec/context.md
}
```

**Cache hit = free.** The AI gets instant project context at zero cost on repeat builds.

**Cache invalidation triggers:**
- `build.gradle.kts` changed (new deps, plugins, config)
- `gradle.properties` changed
- Dependency lock files changed
- Gradle version changed

**NOT invalidated by:**
- Source code changes (architecture doesn't change with every edit)
- Resource changes
- Test changes

## Phases

### Phase 1: Project Context (cacheable) ← START HERE
- `openspecContext` task
- Outputs `.openspec/context.md`
- Project metadata, module graph, dependency tree, source sets
- Plugin/framework detection with architecture hints
- `@CacheableTask` — fast after first run
- Context file auto-added to `.gitignore`

### Phase 2: Smart Templates
- Templates reference `./gradlew openspecContext` instead of `find` / `grep`
- Propose/apply/archive tasks become Gradle-native
- AI calls `./gradlew openspecNew --name=add-auth` instead of `mkdir -p`
- Task outputs feed directly into AI context

### Phase 3: Deep Analysis (optional, expensive)
- Source file scanning (class names, public APIs, endpoint maps)
- Cross-reference: which classes depend on which modules
- "Hot zones" — files changed most in git history
- Test coverage mapping (which tests cover which modules)
- This is expensive → separate task, NOT cached with Phase 1

### Phase 4: Incremental Context Updates
- Track what changed since last context generation
- "Delta context" — only what the AI needs to know about recent changes  
- Ties into the openspec change workflow

## Integration with AI Tools

The context file lives at `.openspec/context.md`. Each tool adapter makes the AI aware of it:

**GitHub Copilot:** Reference in skills as "Read .openspec/context.md first"
**Claude Code:** Add to `.claude/skills/` as a "project understanding" skill

The templates would change from:
```markdown
# Before (bash-heavy)
1. Run `find src -name "*.kt" | head -20` to understand structure
2. Run `cat build.gradle.kts` to see dependencies
3. Figure out the architecture...
```

To:
```markdown
# After (Gradle-powered)
1. Read `.openspec/context.md` for full project context
2. If stale, run `./gradlew openspecContext` to refresh
3. You now know the architecture. Proceed.
```

## Open Questions

1. **JSON vs Markdown?** JSON is machine-parseable, Markdown is AI-readable. Leaning Markdown since the consumer is an LLM, not a program. Could do both.

2. **How deep on dependencies?** Full transitive tree is huge. Top-level + key transitives (Spring Boot starters expanded one level) seems right.

3. **Source scanning in Phase 1?** Tempting but expensive and hard to cache properly (invalidates on any source change). Keep it for Phase 3.

4. **Per-module or single file?** Single file for small projects, per-module files for large monorepos? Start with single file, split later.

5. **Should `openspecSync` auto-run `openspecContext`?** Probably yes — always have fresh context when generating AI files.
