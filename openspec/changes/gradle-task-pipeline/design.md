# Design: Gradle Task Pipeline

## Task Catalog

### Discovery Tasks (read-only, cacheable)

| Task | Output | Purpose |
|---|---|---|
| `openspecContext` | `.openspec/context.md` | Project metadata, plugins, frameworks, git info (**exists**) |
| `openspecTree` | `.openspec/tree.md` | Source tree per module/package/scope |
| `openspecDeps` | `.openspec/deps.md` | Resolved dependencies with versions, classified |
| `openspecModules` | `.openspec/modules.md` | Module graph with boundaries |
| `openspecDevLoop` | `.openspec/devloop.md` | Build/test/run commands per module |
| `openspecStatus` | `.openspec/status.md` | Proposal dashboard + task status |

### Mutation Tasks (write, not cacheable)

| Task | Purpose |
|---|---|
| `openspecSync` | Generate skill/command files for agents (**exists**) |
| `openspecPatch` | Apply a text replacement in a file |
| `openspecScaffold` | Create files from templates |
| `openspecPropose` | Create a proposal with artifacts (**exists**, needs rework) |
| `openspecApply` | Execute a proposal task (**exists**, needs rework) |

## Output Contract

Every discovery task follows:

```kotlin
@CacheableTask
abstract class OpenSpecTreeTask : DefaultTask() {
    @get:InputFiles @get:PathSensitive(PathSensitivity.RELATIVE)
    abstract val sourceFiles: ConfigurableFileCollection

    @get:OutputFile
    abstract val outputFile: RegularFileProperty  // .openspec/tree.md

    @TaskAction
    fun generate() {
        // Build the Markdown content
        // Write to outputFile
    }
}
```

**Rules:**
- Output is always a single `.openspec/*.md` file
- Content is Markdown — human-readable AND AI-parseable
- Files are small and focused (one concern per file)
- Cacheable via Gradle's input/output tracking
- Re-generates only when inputs change

## Scoping via Task Options

Tasks accept `--module`, `--scope`, `--package` options where relevant:

```bash
./gradlew openspecTree                          # full project
./gradlew openspecTree --module=app             # just :app
./gradlew openspecTree --module=app --scope=main  # just :app main sources
```

Options use Gradle's `@Option` annotation:

```kotlin
@get:Input @get:Optional
@get:Option(option = "module", description = "Target module")
abstract val module: Property<String>
```

## Skill/Prompt Template Updates

### Before (bash-dependent)
```markdown
## Understanding the Project
Run these commands to understand the codebase:
- `find src/ -name '*.kt' | head -50` — source files
- `cat build.gradle.kts` — build config
- `grep -r "implementation" build.gradle.kts` — dependencies
```

### After (task-dependent)
```markdown
## Understanding the Project
Read these files (generated by Gradle — run `./gradlew openspecSync` if missing):
- `.openspec/context.md` — project config, plugins, frameworks
- `.openspec/tree.md` — source layout
- `.openspec/deps.md` — dependency list with versions

To regenerate: `./gradlew openspecContext openspecTree openspecDeps`
```

## Task Description Convention: Descriptions as Tool Schema

Every task's `description` field doubles as an AI-readable tool definition. Convention:

```kotlin
tasks.register("openspecTree", OpenSpecTreeTask::class.java) {
    group = "openspec"
    description = buildString {
        appendLine("[tool] Source tree generator.")
        appendLine("Output: .openspec/tree.md")
        appendLine("Options: --module=:app --scope=main|test --package=com.example.ui")
        appendLine("Use when: You need to understand file layout, find classes, or navigate the codebase.")
        appendLine("Chain: Read output → openspecDeps for dependency context.")
    }
}
```

### Schema Fields (embedded in description)

| Field | Purpose | Example |
|---|---|---|
| `[tool]` | Prefix — signals this is a queryable tool | `[tool] Source tree generator.` |
| `Output:` | Where the result file lives | `Output: .openspec/tree.md` |
| `Options:` | Available parameters | `Options: --module=:app --scope=main\|test` |
| `Use when:` | Intent matching — tells AI when to pick this tool | `Use when: You need dependency versions` |
| `Chain:` | Pipeline hint — what to run next | `Chain: Read output → openspecDeps` |

### Why This Works

1. `./gradlew tasks --group openspec` = **tool catalog** for any AI agent
2. `./gradlew help --task openspecTree` = **tool docs** with `@Option` details
3. No custom metadata format — pure Gradle native
4. Human-readable AND machine-parseable
5. Works with every AI tool (Copilot, Claude, Cursor, Codex) — they all can run Gradle

## Chaining: Task Output → AI Decision → Next Task

```
Agent reads .openspec/context.md
  → sees "uses Hilt, Compose, Room"
  → decides to check deps for versions
Agent runs: ./gradlew openspecDeps
  → reads .openspec/deps.md
  → sees "hilt:2.48 — migration hint: consider 2.51"
  → decides to propose an upgrade
Agent runs: ./gradlew openspecPropose --name=upgrade-hilt
  → creates openspec/changes/upgrade-hilt/
  → reads proposal, starts applying
```

No bash at any step. Gradle handles all filesystem access.

## Auto-regeneration

Discovery tasks are cacheable. When source files change, running `openspecSync` can depend on all discovery tasks:

```kotlin
tasks.named("openspecSync") {
    dependsOn("openspecContext", "openspecTree", "openspecDeps", "openspecStatus")
}
```

One command refreshes everything: `./gradlew openspecSync`

Or individual tasks for targeted updates.

## File Size Budget

Each `.openspec/*.md` file should stay under ~4KB for token efficiency:
- `context.md` — project metadata: ~2-3KB ✅
- `tree.md` — source listing: ~1-4KB (scoped by module)
- `deps.md` — dependency table: ~1-3KB
- `modules.md` — module graph: ~0.5-2KB
- `devloop.md` — commands: ~0.5-1KB
- `status.md` — dashboard: ~1-2KB

Total context budget for AI: ~10-15KB across all files. Much smaller than a single `find` dump.
